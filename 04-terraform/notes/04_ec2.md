# 04 EC2 Preparation

このメモでは、TerraformでEC2作成に進む前の準備として追加したKey PairとAMI取得について整理する。

Security Groupまで作成できたため、次はBastion EC2とWeb EC2をTerraform化する。

その前提として、以下を先に用意した。

```text
aws_key_pair
data "aws_ami"
locals.web_ami_id
```

今回の `terraform plan` では、Security Groupまでの37リソースにKey Pairが1つ増え、以下を確認した。

```text
Plan: 38 to add, 0 to change, 0 to destroy.
```

`data "aws_ami"` と `locals` はAWSリソースを作成するものではないため、`to add` の数には含まれない。

## この段階のゴール

EC2本体を作る前に、以下をTerraformで表現する。

```text
SSH用Key Pair
Bastion用Amazon Linux 2023 AMI取得
Web EC2用AMIの切り替え準備
```

EC2本体は次の段階で作成する。

## Key Pairの考え方

EC2へSSHするにはKey Pairが必要である。

AWS CLI編では、日次ラボ用に `nobu.pem` を作成し、cleanup時に削除する運用にしていた。

Terraformでは、秘密鍵をTerraformで生成するのではなく、Mac上に既にある公開鍵をAWSへ登録する方針にした。

理由:

- 秘密鍵をTerraform stateに含めたくない
- GitHubへ秘密鍵を誤って上げるリスクを避けたい
- Terraformでは公開鍵だけをAWS Key Pairとして登録すればよい

今回の考え方:

```text
Mac上の秘密鍵:
  ~/.ssh/id_ed25519

Mac上の公開鍵:
  ~/.ssh/id_ed25519.pub

AWS上のKey Pair名:
  nobu
```

Terraformは公開鍵 `~/.ssh/id_ed25519.pub` の内容を読み込み、AWSに `nobu` というKey Pairを作成する。

秘密鍵そのものはAWSにもGitHubにも渡さない。

## variables.tf

Key Pair関連の変数を追加した。

```hcl
variable "key_pair_name" {
  description = "EC2 key pair name used for SSH."
  type        = string
  default     = "nobu"
}
```

`key_pair_name` はAWS上に作成するKey Pair名である。

```hcl
variable "public_key_path" {
  description = "Path to the public key file registered as EC2 key pair."
  type        = string
  default     = "~/.ssh/id_ed25519.pub"
}
```

`public_key_path` はTerraformが読み込む公開鍵ファイルのパスである。

もし `~` の展開でエラーになる場合は、絶対パスにする。

```hcl
default = "/Users/nobu/.ssh/id_ed25519.pub"
```

Web EC2用のAMI切り替えに備えて、以下も追加した。

```hcl
variable "use_custom_web_ami" {
  description = "Whether to use custom AMI for web servers."
  type        = bool
  default     = false
}
```

```hcl
variable "custom_web_ami_id" {
  description = "Custom AMI ID for web servers. Used when use_custom_web_ami is true."
  type        = string
  default     = ""
}
```

`use_custom_web_ami = true` にした場合は、Ruby、nginx、deploy userなどを事前導入したカスタムAMIをWeb EC2に使う想定である。

## aws_key_pair

Terraformでは `aws_key_pair` resourceでAWS Key Pairを作成する。

```hcl
resource "aws_key_pair" "main" {
  key_name   = var.key_pair_name
  public_key = file(var.public_key_path)

  tags = merge(local.common_tags, {
    Name = var.key_pair_name
  })
}
```

重要なのは以下。

```hcl
public_key = file(var.public_key_path)
```

`file()` は、指定したローカルファイルの内容を読み込むTerraform関数である。

ここでは公開鍵ファイルを読み込み、AWS Key Pairへ登録している。

秘密鍵ではなく公開鍵を指定する点に注意する。

```text
秘密鍵: ~/.ssh/id_ed25519
公開鍵: ~/.ssh/id_ed25519.pub
```

Terraformで読むのは公開鍵だけである。

## 公開鍵の確認

今回、Mac側には以下の公開鍵が存在していた。

```bash
ls -l ~/.ssh/id_ed25519.pub
```

`~/.ssh/nobu.pub` は存在しなかったため、`public_key_path` は以下に変更した。

```hcl
default = "~/.ssh/id_ed25519.pub"
```

もし公開鍵がない場合は、秘密鍵から作成できる。

```bash
ssh-keygen -y -f ~/.ssh/id_ed25519 > ~/.ssh/id_ed25519.pub
```

ただし、秘密鍵の扱いには注意する。

## data "aws_ami"

Bastion EC2ではAmazon Linux 2023の最新AMIを使う。

Terraformでは `data "aws_ami"` で既存AMIを検索できる。

```hcl
data "aws_ami" "amazon_linux_2023" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }

  filter {
    name   = "architecture"
    values = ["x86_64"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }

  filter {
    name   = "root-device-type"
    values = ["ebs"]
  }
}
```

ポイント:

| 項目 | 意味 |
| :--- | :--- |
| `most_recent = true` | 条件に合う最新AMIを取得する |
| `owners = ["amazon"]` | Amazon公式AMIに限定する |
| `name` filter | Amazon Linux 2023のAMI名に限定する |
| `architecture` | x86_64に限定する |
| `virtualization-type` | HVMに限定する |
| `root-device-type` | EBS rootに限定する |

`data` は既存情報を取得するだけなので、AWS上に新しいリソースは作らない。

そのため、`terraform plan` の `to add` には含まれない。

## Web EC2用AMIの切り替え

Web EC2は、標準のAmazon Linux 2023 AMIを使う方法と、事前にRubyやnginxを入れたカスタムAMIを使う方法がある。

毎回Rubyをソースビルドすると時間がかかるため、学習効率を上げるにはカスタムAMIが有効である。

Terraformでは、以下のlocal値で切り替える。

```hcl
locals {
  web_ami_id = var.use_custom_web_ami ? var.custom_web_ami_id : data.aws_ami.amazon_linux_2023.id
}
```

意味:

```text
use_custom_web_ami が true
  -> custom_web_ami_id を使う

use_custom_web_ami が false
  -> Amazon Linux 2023の最新AMIを使う
```

Terraformの三項演算子:

```hcl
condition ? trueの場合の値 : falseの場合の値
```

今回の場合:

```hcl
var.use_custom_web_ami ? var.custom_web_ami_id : data.aws_ami.amazon_linux_2023.id
```

## main.tfとlocals.tfの整理

`locals` は複数ファイルに分かれていてもTerraform上は問題ない。

ただし、読みやすさを優先するなら、既存の `locals.tf` にまとめるとよい。

例:

```hcl
locals {
  common_tags = {
    Project     = "terraform-iac-lab"
    Environment = "learning"
  }

  web_ami_id = var.use_custom_web_ami ? var.custom_web_ami_id : data.aws_ami.amazon_linux_2023.id
}
```

必須ではないが、プロジェクトが大きくなるほど、値の置き場所を揃えると読みやすい。

## よくあるミス

### 公開鍵パスを間違える

誤り:

```hcl
default = "~/.ssh/nobu/pub"
```

これは `~/.ssh/nobu/` ディレクトリ配下の `pub` ファイルを意味してしまう。

正しい例:

```hcl
default = "~/.ssh/id_ed25519.pub"
```

または、

```hcl
default = "/Users/nobu/.ssh/id_ed25519.pub"
```

### 秘密鍵を指定してしまう

誤り:

```hcl
public_key = file("~/.ssh/id_ed25519")
```

正しい:

```hcl
public_key = file("~/.ssh/id_ed25519.pub")
```

AWS Key Pairに登録するのは公開鍵である。

秘密鍵はローカルにだけ保持する。

### `key_pair_name` と `public_key_path` を混同する

`key_pair_name` はAWS上のKey Pair名。

```hcl
key_pair_name = "nobu"
```

`public_key_path` はローカルの公開鍵ファイルパス。

```hcl
public_key_path = "~/.ssh/id_ed25519.pub"
```

意味:

```text
~/.ssh/id_ed25519.pub の中身を使って、
AWSに nobu というKey Pairを登録する
```

## セキュリティ注意

以下はGitHubに上げない。

```text
秘密鍵
terraform.tfvars
terraform.tfstate
terraform.tfstate.backup
.terraform/
```

特に、`terraform.tfvars` に以下のような値を書く場合は注意する。

```hcl
admin_ip_cidr = "x.x.x.x/32"
custom_web_ami_id = "ami-xxxxxxxxxxxxxxxxx"
```

AMI ID自体は秘密情報ではないが、環境情報を不用意に出したくない場合は `terraform.tfvars` 側に逃がす。

`terraform.tfstate` には作成したAWSリソース情報が入るため、GitHubへ上げない。
